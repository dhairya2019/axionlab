  <action>
Create `app/insights/[slug]/` directory and `page.tsx` inside it.

This file has four critical exports:

1. **`dynamicParams = false`** — MUST be set. Forces Next.js to return 404 for slugs not in generateStaticParams output. Without this, unknown slugs attempt on-demand rendering and fail at runtime.

2. **`generateStaticParams()`** — reads all post slugs from lib/blog.ts at build time. Returns `{ slug: string }[]` for static pre-rendering.

3. **`generateMetadata()`** — reads post metadata from lib/blog.ts for SEO. Returns title, description, openGraph (type: article, publishedTime, tags), and twitter card metadata. CRITICAL: `params` is a `Promise` in Next.js 16 — must `await` before accessing `slug`.

4. **Default export page component** — async Server Component that:
   - Awaits params to get slug (Next.js 16 breaking change)
   - Reads post metadata from lib/blog.ts
   - Dynamically imports the MDX file: `await import(`@/content/blog/${slug}.mdx`)`
   - Renders a post header (title, date, reading time, tags) followed by the MDX content wrapped in `prose prose-invert max-w-none`

```typescript
import type { Metadata } from 'next'
import { getAllPosts, getPostBySlug } from '@/lib/blog'

export const dynamicParams = false

export async function generateStaticParams() {
  const posts = getAllPosts()
  return posts.map(post => ({ slug: post.slug }))
}

export async function generateMetadata({
  params,
}: {
  params: Promise<{ slug: string }>
}): Promise<Metadata> {
  const { slug } = await params
  const post = getPostBySlug(slug)

  return {
    title: `${post.title} | AXIONLAB Insights`,
    description: post.description,
    openGraph: {
      title: post.title,
      description: post.description,
      type: 'article',
      publishedTime: post.date,
      tags: post.tags,
    },
    twitter: {
      card: 'summary_large_image',
      title: post.title,
      description: post.description,
    },
  }
}

export default async function BlogPost({
  params,
}: {
  params: Promise<{ slug: string }>
}) {
  const { slug } = await params
  const post = getPostBySlug(slug)
  const { default: Post } = await import(`@/content/blog/${slug}.mdx`)

  return (
    <article className="max-w-3xl mx-auto px-6 pt-40 pb-40">
      {/* Post header */}
      <header className="mb-16 border-t border-white/10 pt-8">
        <p className="text-[10px] text-accent font-black uppercase tracking-[0.6em] mb-6">
          {post.date} — {post.readingTime}
        </p>
        <h1 className="font-condensed text-5xl md:text-7xl font-black uppercase tracking-tighter leading-none mb-8">
          {post.title}
        </h1>
        <div className="flex gap-3 flex-wrap">
          {post.tags.map(tag => (
            <span key={tag} className="text-[9px] font-black uppercase tracking-[0.4em] text-muted border border-white/10 px-3 py-1">
              {tag}
            </span>
          ))}
        </div>
      </header>
      {/* MDX prose content */}
      <div className="prose prose-invert max-w-none">
        <Post />
      </div>
    </article>
  )
}
```

Key implementation details:
- `params` is `Promise<{ slug: string }>` — MUST await. This is a Next.js 16 breaking change. Accessing `params.slug` directly returns undefined.
- `await import(`@/content/blog/${slug}.mdx`)` — dynamic import loads the compiled MDX React component. The `@/` path alias maps to the project root via tsconfig.json.
- `prose prose-invert max-w-none` — applies Tailwind typography. `prose-invert` activates dark mode base. The --tw-prose-* variables in globals.css override to AXIONLAB colors. `max-w-none` prevents prose from constraining width (article container handles it).
- `max-w-3xl` on article container — 768px max width for comfortable reading line length.
- `pt-40` — clears the fixed Nav component (same as listing page).
- Post header design matches the pattern from the research doc (Pattern 2): date + reading time at top, large title, tags below.
- mdx-components.tsx overrides are applied automatically by @next/mdx — no explicit import needed.
- rehype-pretty-code syntax highlighting is applied at build time in the MDX compilation pipeline — zero client JS.
