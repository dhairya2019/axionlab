**Step 1: Add `getAdjacentPosts` function to lib/blog.ts**

Add this function after the existing `getPostBySlug` function:

```typescript
export function getAdjacentPosts(
  slug: string,
  posts?: PostMeta[]
): { prev: PostMeta | null; next: PostMeta | null } {
  const allPosts = posts ?? getAllPosts()
  const index = allPosts.findIndex(p => p.slug === slug)

  if (index === -1) return { prev: null, next: null }

  // allPosts is sorted newest-first:
  // index + 1 = older post (previous)
  // index - 1 = newer post (next)
  return {
    prev: index < allPosts.length - 1 ? allPosts[index + 1] : null,
    next: index > 0 ? allPosts[index - 1] : null,
  }
}
```

Key details:
- Accepts optional `posts` parameter to reuse pre-fetched array (avoids double filesystem read)
- Falls back to `getAllPosts()` if not provided
- Returns `null` for boundary posts (first/last have no next/prev respectively)
- "prev" = older (higher index in newest-first), "next" = newer (lower index)

**Step 2: Create components/blog/PostNavigation.tsx**

```typescript
import Link from 'next/link'
import type { PostMeta } from '@/lib/blog'

export function PostNavigation({
  prev,
  next,
}: {
  prev: PostMeta | null
  next: PostMeta | null
}) {
  if (!prev && !next) return null

  return (
    <nav className="border-t border-white/10 pt-12 mt-16 grid grid-cols-2 gap-8" aria-label="Post navigation">
      <div>
        {prev && (
          <Link href={`/insights/${prev.slug}`} className="group block">
            <p className="text-[9px] font-black uppercase tracking-[0.4em] text-muted mb-2">
              Previous
            </p>
            <p className="text-sm font-black uppercase tracking-tight group-hover:text-accent transition-colors leading-tight">
              {prev.title}
            </p>
          </Link>
        )}
      </div>
      <div className="text-right">
        {next && (
          <Link href={`/insights/${next.slug}`} className="group block">
            <p className="text-[9px] font-black uppercase tracking-[0.4em] text-muted mb-2">
              Next
            </p>
            <p className="text-sm font-black uppercase tracking-tight group-hover:text-accent transition-colors leading-tight">
              {next.title}
            </p>
          </Link>
        )}
      </div>
    </nav>
  )
}
```

- No border-radius, no box-shadow
- `border-t border-white/10` separator matches existing page patterns
- `text-[9px] font-black uppercase tracking-[0.4em] text-muted` — micro-label pattern used site-wide
- `group-hover:text-accent` — hover color matches PostCard pattern
- Server Component — no 'use client' needed, `next/link` works server-side

## The Latency Problem

Most API performance issues trace back to architectural decisions made in the first week. By the time latency becomes visible, the patterns are baked into the codebase.

### Connection Pooling

Database connections are expensive to establish. A well-configured pool eliminates this overhead:

```typescript
const pool = createPool({
  max: 20,
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 2000,
})
```

### Response Compression

Every byte matters at scale. Enable compression at the edge, not the origin:

> The fastest code is the code that never runs. The fastest bytes are the bytes that never transfer.

### Caching Strategies

Layer your caches: CDN → application → database query cache. Each layer absorbs a different access pattern.

- **CDN**: Static responses, TTL-based invalidation
- **Application**: Computed results, LRU eviction
- **Query cache**: Parameterized queries, write-through invalidation
```

## Instrument First, Build Second

The conventional approach — build features, then add monitoring — creates blind spots. By the time you add observability, the system's failure modes are already invisible.

### Structured Logging

Every log entry should be machine-parseable:

```typescript
logger.info('request_processed', {
  duration_ms: 42,
  status: 200,
  endpoint: '/api/orders',
  trace_id: context.traceId,
})
```

### Metrics That Matter

Not all metrics are equal. Focus on the four golden signals:

- **Latency**: Time to serve a request
- **Traffic**: Demand on the system
- **Errors**: Rate of failed requests
- **Saturation**: How full the system is

> You cannot improve what you cannot measure. But measuring everything is just noise.

### Distributed Tracing

In a microservices architecture, a single user request touches dozens of services. Without tracing, debugging is archaeology.
