Selling a product that is not in stock is the most expensive bug in commerce. It triggers a refund workflow, a customer support interaction, a trust deficit, and a negative review. At scale, overselling a single SKU during a flash sale can cascade into thousands of failed orders within seconds.

Real-time inventory management is not a feature -- it is infrastructure. It sits at the intersection of distributed systems, concurrent data access, and domain-specific consistency requirements. Getting it wrong costs revenue. Getting it right requires engineering discipline that most teams underestimate.

## The Inventory Consistency Problem

Inventory appears simple: a number goes up when stock arrives, down when stock sells. But in a distributed commerce system, that number is being read and written by dozens of concurrent processes.

Consider a single SKU with 5 units in stock. Three customers add it to their carts simultaneously. Two warehouse workers scan incoming shipments. A marketplace integration pushes an external order. A returns processor adds one unit back. All of these operations hit the inventory count within the same second.

Without careful engineering, the final count is wrong. And the system does not know it is wrong.

### Why Traditional CRUD Fails

A naive implementation reads the current count, modifies it, and writes it back:

```typescript
// This is broken under concurrency
async function purchaseItem(skuId: string, quantity: number) {
  const item = await db.inventory.findUnique({ where: { skuId } })
  if (item.quantity >= quantity) {
    await db.inventory.update({
      where: { skuId },
      data: { quantity: item.quantity - quantity },
    })
    return { success: true }
  }
  return { success: false, reason: 'out_of_stock' }
}
```

Between the `findUnique` and `update` calls, another transaction can read the same stale count and also succeed. Two customers each buy the last 3 units of a 5-unit SKU, and the system records -1 units in stock.

The fix is atomic operations:

```typescript
async function purchaseItem(skuId: string, quantity: number) {
  const result = await db.inventory.updateMany({
    where: {
      skuId,
      quantity: { gte: quantity },
    },
    data: {
      quantity: { decrement: quantity },
    },
  })
  return { success: result.count > 0 }
}
```

This is better -- the database handles atomicity. But it only solves the single-database, single-warehouse case. Real commerce systems are more complex.

## Event Sourcing for Inventory

Instead of storing a mutable count, store every event that changes the count. The current quantity is a derived value computed by replaying the event log.

```typescript
type InventoryEvent =
  | { type: 'STOCK_RECEIVED'; skuId: string; quantity: number; warehouseId: string }
  | { type: 'STOCK_RESERVED'; skuId: string; quantity: number; orderId: string }
  | { type: 'STOCK_COMMITTED'; skuId: string; quantity: number; orderId: string }
  | { type: 'STOCK_RELEASED'; skuId: string; quantity: number; orderId: string }
  | { type: 'STOCK_RETURNED'; skuId: string; quantity: number; returnId: string }
  | { type: 'STOCK_ADJUSTED'; skuId: string; quantity: number; reason: string }
```

### Why Events Over State

Events are append-only. They never conflict. Two warehouses can independently record `STOCK_RECEIVED` events without coordination. The materialized view reconciles them.

Events are auditable. When the count is wrong -- and it will be wrong eventually -- the event log tells you exactly what happened and when. Debugging a mutable integer column in a SQL table at 3am during a sale event is a different kind of miserable.

Events enable temporal queries. "What was the inventory position at 14:32 UTC?" is answered by replaying events up to that timestamp. Try doing that with a mutable count.

### Materialized Views

Raw event replay is too slow for real-time stock checks. Materialized views maintain pre-computed state:

```typescript
interface InventoryPosition {
  skuId: string
  warehouseId: string
  available: number    // physically present and unreserved
  reserved: number     // held for pending orders
  incoming: number     // in transit from suppliers
  updatedAt: Date
}

function applyEvent(position: InventoryPosition, event: InventoryEvent): InventoryPosition {
  switch (event.type) {
    case 'STOCK_RECEIVED':
      return { ...position, available: position.available + event.quantity }
    case 'STOCK_RESERVED':
      return {
        ...position,
        available: position.available - event.quantity,
        reserved: position.reserved + event.quantity,
      }
    case 'STOCK_COMMITTED':
      return { ...position, reserved: position.reserved - event.quantity }
    case 'STOCK_RELEASED':
      return {
        ...position,
        available: position.available + event.quantity,
        reserved: position.reserved - event.quantity,
      }
    default:
      return position
  }
}
```

The materialized view updates asynchronously as events are appended. If the view crashes, it rebuilds from the event log. No data is lost.

## CQRS: Separating Reads from Writes

Command Query Responsibility Segregation splits the inventory system into two distinct paths.

### The Command Side

Handles writes: reserve stock, receive shipment, adjust count. Each command validates business rules and emits events.

```typescript
async function handleReserveStock(command: ReserveStockCommand): Promise<Result> {
  const position = await readModel.getPosition(command.skuId, command.warehouseId)

  if (position.available < command.quantity) {
    return { success: false, reason: 'insufficient_stock' }
  }

  await eventStore.append({
    type: 'STOCK_RESERVED',
    skuId: command.skuId,
    quantity: command.quantity,
    orderId: command.orderId,
    timestamp: Date.now(),
  })

  return { success: true, reservationId: generateId() }
}
```

### The Query Side

Handles reads: check stock availability, get warehouse levels, generate reports. The query side reads from optimized projections, not the event log.

```typescript
// Optimized for storefront display -- fast reads, eventual consistency
async function getStockLevel(skuId: string): Promise<StockDisplay> {
  const positions = await cache.get(`stock:${skuId}`)
  if (positions) return positions

  const result = await readDb.query(
    'SELECT SUM(available) as total FROM inventory_positions WHERE sku_id = $1',
    [skuId]
  )

  const display = {
    available: result.rows[0].total,
    label: result.rows[0].total > 10 ? 'In Stock' : `Only ${result.rows[0].total} left`,
  }

  await cache.set(`stock:${skuId}`, display, { ttl: 5 })
  return display
}
```

The read path can be scaled independently. Add read replicas, cache layers, and CDN-level stock indicators without touching the write path. This separation is critical during high-traffic events where read volume can spike 100x while write volume only spikes 10x.

## Distributed Warehouse Synchronization

Enterprise commerce operates across multiple fulfillment centers. Each warehouse has its own physical stock, and the storefront must present a unified availability view.

### The Synchronization Challenge

Warehouse A has 50 units. Warehouse B has 30 units. The storefront shows 80 available. A customer orders 60 units. Which warehouse fulfills it? What if Warehouse A just shipped 20 units to another order that has not been recorded yet?

### Event-Driven Sync Protocol

Each warehouse publishes events to a central event bus. The aggregation service consumes events from all warehouses and maintains a global position:

```typescript
interface WarehouseSync {
  globalPosition: Map<string, number>  // skuId -> total available
  warehousePositions: Map<string, Map<string, number>>  // warehouseId -> skuId -> available
  lastSyncTimestamp: Map<string, number>  // warehouseId -> last event timestamp
}

function detectStaleWarehouse(sync: WarehouseSync, thresholdMs: number): string[] {
  const now = Date.now()
  const stale: string[] = []
  for (const [warehouseId, lastSync] of sync.lastSyncTimestamp) {
    if (now - lastSync > thresholdMs) {
      stale.push(warehouseId)
    }
  }
  return stale
}
```

When a warehouse goes stale (no events for a configurable threshold), the system can either exclude its inventory from the global count or flag it for manual reconciliation. Both are valid strategies depending on the business context.

## Conflict Resolution and Race Conditions

Flash sales are the stress test every inventory system faces. Thousands of concurrent purchase attempts on limited stock within seconds. The system must be both fast and correct.

### The Reservation Pattern

Instead of directly decrementing stock on purchase, use a two-phase reservation:

1. **Reserve** -- atomically decrement available stock, increment reserved stock
2. **Commit** -- on successful payment, decrement reserved stock (item ships)
3. **Release** -- on payment failure or timeout, decrement reserved, increment available

```typescript
// Reservation with automatic expiry
async function createReservation(skuId: string, quantity: number, orderId: string) {
  const reserved = await atomicReserve(skuId, quantity)
  if (!reserved) return { success: false }

  // Schedule automatic release if not committed within 10 minutes
  await scheduler.enqueue({
    type: 'RELEASE_RESERVATION',
    skuId,
    quantity,
    orderId,
    executeAt: Date.now() + 10 * 60 * 1000,
  })

  return { success: true, expiresIn: 600 }
}
```

The automatic release prevents ghost reservations -- carts that were abandoned, payment pages that timed out, or API clients that crashed. Without this, stock becomes permanently locked.

### Handling Oversell at the Database Level

For the critical atomic reserve operation, use database-level constraints:

```sql
UPDATE inventory_positions
SET
  available = available - $1,
  reserved = reserved + $1,
  updated_at = NOW()
WHERE
  sku_id = $2
  AND warehouse_id = $3
  AND available >= $1
RETURNING available, reserved;
```

The `WHERE available >= $1` clause is the guard. If two transactions race, only one will find sufficient stock. The other gets zero rows affected and returns an out-of-stock response. The database serializes the conflict. No application-level locking is needed.

### Distributed Lock Fallback

For operations that span multiple database rows or services, a distributed lock may be necessary:

```typescript
async function fulfillMultiWarehouseOrder(items: OrderItem[]) {
  const lockKey = `fulfill:${items.map(i => i.skuId).sort().join(':')}`
  const lock = await redlock.acquire([lockKey], 5000)

  try {
    // Check availability across warehouses
    // Allocate from optimal locations
    // Reserve in a single transaction
    await allocateAcrossWarehouses(items)
  } finally {
    await lock.release()
  }
}
```

Use distributed locks sparingly. They serialize operations and become bottlenecks at scale. Prefer the atomic database approach for single-SKU operations.

## Monitoring and Alerting

An inventory system without monitoring is a liability. You need to detect problems before customers do.

### Critical Metrics

- **Stock accuracy rate**: periodic physical counts compared to system counts (target: 99.5%+)
- **Reservation timeout rate**: percentage of reservations that expire without committing (indicates checkout friction)
- **Event processing lag**: delay between event emission and materialized view update (target: under 500ms)
- **Oversell incidents**: count of orders accepted for out-of-stock items (target: zero)
- **Sync staleness**: time since last event from each warehouse

```typescript
// Health check endpoint for inventory system
async function inventoryHealthCheck(): Promise<HealthStatus> {
  const lag = await getEventProcessingLag()
  const staleWarehouses = detectStaleWarehouse(syncState, 60_000)
  const recentOversells = await countOversellsInWindow(3600_000)

  return {
    status: lag < 1000 && staleWarehouses.length === 0 && recentOversells === 0
      ? 'healthy'
      : 'degraded',
    metrics: {
      eventLagMs: lag,
      staleWarehouses,
      recentOversells,
    },
  }
}
```

### Alerting Thresholds

Set alerts at business-meaningful thresholds, not arbitrary numbers:

- **Event lag exceeds 2 seconds**: storefront may show stale availability
- **Any warehouse stale for 5 minutes**: exclude from global count or investigate
- **Oversell count exceeds 0**: immediate investigation, potential halt on affected SKUs
- **Reservation timeout rate exceeds 15%**: checkout flow has a problem

## Architecture Principles

Building inventory infrastructure that scales requires commitment to a few non-negotiable principles:

1. **Events are the source of truth** -- mutable state is derived, never primary
2. **Atomic operations at the boundary** -- the database enforces consistency, not application code
3. **Reservations expire** -- every held resource must have a timeout
4. **Monitor everything** -- if you cannot measure it, you cannot trust it
5. **Design for the spike** -- steady-state performance is irrelevant if the system fails during peak demand

> The difference between a commerce platform that handles flash sales and one that collapses under them is not raw throughput. It is the consistency model.
